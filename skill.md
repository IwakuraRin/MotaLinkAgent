# 代码格式与注释规范 Skill

本文件用于约束本仓库后续所有代码的书写格式。无论是前端、后端、ROS、固件、脚本还是配置生成代码，都应按同一套结构组织：先说明代码作用，再按功能模块分块，相关功能放在一起，注释使用中文。

## 总原则

1. 每个代码文件开头必须用简短中文注释说明“下面代码的作用”。
2. 代码必须模块化，一个文件只承担清晰的职责，不把无关功能混在一起。
3. 一个功能的代码必须用中文注释框起来，注释里写明这段代码是干什么的。
4. 同类功能必须集中放置，例如按钮、滑块、输入框、状态、事件处理、接口请求分别成块。
5. 通用 API 放到文件最后或统一的公共 API 文件中；某个模块专用 API 放在对应模块内部。
6. 注释用于解释模块职责和业务意图，不写空泛注释，不逐行解释显而易见的语法。
7. 文件名、目录名、函数名要让人一眼看出用途，避免 `utils2`、`test_new`、`handleData` 这类模糊命名。

## 推荐文件结构

每个代码文件按下面顺序组织：

1. 文件开头说明
2. 依赖导入
3. 类型、常量、配置
4. 当前模块状态或数据结构
5. UI 控件或外部入口
6. 功能逻辑
7. 事件处理
8. 模块专用 API
9. 通用 API 或导出

## 注释框格式

推荐统一使用下面这种注释框。不同语言可以按语法调整，但中文说明必须保留。

```ts
// ==================== 按钮控件 ====================
// 作用：集中定义本页面所有按钮的显示、禁用状态和点击入口。
// ==================================================
```

```html
<!-- ==================== 按钮控件 ==================== -->
<!-- 作用：集中渲染本页面所有可点击按钮。 -->
<!-- ================================================== -->
```

```py
# ==================== 运动控制 ====================
# 作用：封装机器人底盘速度、方向和停止控制。
# ==================================================
```

## 前端代码规范

前端文件开头必须说明页面、组件或工具函数作用。Vue、React、React Native、普通 TypeScript 都按这个顺序组织：

1. 页面/组件说明
2. 依赖导入
3. 类型定义
4. 常量配置
5. 按钮、滑块、输入框、开关等控件配置
6. 页面状态
7. 计算属性或派生数据
8. 点击、拖动、输入等事件处理
9. 页面功能函数
10. 当前模块 API
11. 通用 API

示例：

```ts
// 作用：机器人控制页面，负责显示连接状态、运动按钮、速度滑块和急停操作。

import { ref } from "vue";

// ==================== 类型定义 ====================
// 作用：定义控制页面内部使用的数据结构。
// ==================================================
type MoveDirection = "forward" | "backward" | "left" | "right" | "stop";

// ==================== 控件配置 ====================
// 作用：集中管理按钮、滑块等 UI 控件的默认值和显示文案。
// ==================================================
const speedSliderConfig = {
  min: 0,
  max: 100,
  step: 5,
};

// ==================== 页面状态 ====================
// 作用：保存当前连接状态、速度值和最后一次控制指令。
// ==================================================
const isConnected = ref(false);
const speedValue = ref(30);
const lastDirection = ref<MoveDirection>("stop");

// ==================== 按钮事件 ====================
// 作用：处理页面按钮点击，并调用对应的机器人控制能力。
// ==================================================
function handleMoveButtonPress(direction: MoveDirection) {
  lastDirection.value = direction;
  sendMoveCommand(direction, speedValue.value);
}

// ==================== 滑块事件 ====================
// 作用：处理速度滑块变化，并同步当前控制速度。
// ==================================================
function handleSpeedChange(nextSpeed: number) {
  speedValue.value = nextSpeed;
}

// ==================== 机器人控制 API ====================
// 作用：封装当前页面专用的机器人运动控制请求。
// ==================================================
async function sendMoveCommand(direction: MoveDirection, speed: number) {
  return fetch("/api/robot/move", {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({ direction, speed }),
  });
}
```

## 后端代码规范

后端文件必须按“路由入口、参数校验、业务逻辑、数据访问、通用工具”的顺序组织。一个接口的校验、处理和专用 API 应靠近放置，跨模块通用能力放到公共目录。

示例：

```go
// 作用：机器人运动控制接口，负责接收前端指令并转发给底盘控制服务。

package robot

// ==================== 请求结构 ====================
// 作用：定义运动控制接口接收的参数。
// ==================================================
type MoveRequest struct {
    Direction string `json:"direction"`
    Speed     int    `json:"speed"`
}

// ==================== 路由入口 ====================
// 作用：注册机器人控制相关 HTTP 接口。
// ==================================================
func RegisterRobotRoutes(server *Server) {
    server.Post("/api/robot/move", HandleMoveRobot)
}

// ==================== 接口处理 ====================
// 作用：校验请求参数，并调用底盘控制服务。
// ==================================================
func HandleMoveRobot(ctx *Context) error {
    var req MoveRequest
    if err := ctx.Bind(&req); err != nil {
        return err
    }

    return SendMoveCommand(req.Direction, req.Speed)
}

// ==================== 模块专用 API ====================
// 作用：封装机器人运动控制模块自己的底层调用。
// ==================================================
func SendMoveCommand(direction string, speed int) error {
    return nil
}
```

## ROS / Python 代码规范

ROS 节点、Python 脚本必须按“节点说明、参数读取、发布订阅、核心逻辑、设备接口、启动入口”组织。订阅回调、发布逻辑、串口读写、模型推理等必须分块。

## 固件 / Arduino / C++ 代码规范

固件代码必须把引脚定义、通信协议、电机控制、传感器读取、主循环分开。中断、串口解析、I2C 控制等必须单独成块，避免全部塞进 `loop()`。

示例：

```cpp
// 作用：机器人底盘控制固件，负责解析串口指令并控制电机驱动板。

#include <Arduino.h>

// ==================== 引脚配置 ====================
// 作用：集中定义电机、传感器和通信接口使用的硬件引脚。
// ==================================================
const int STATUS_LED_PIN = 13;

// ==================== 通信协议 ====================
// 作用：解析上位机发送的运动控制指令。
// ==================================================
void handleSerialCommand() {
  if (!Serial.available()) {
    return;
  }
}

// ==================== 电机控制 ====================
// 作用：封装底盘电机速度和方向控制。
// ==================================================
void setMotorSpeed(int leftSpeed, int rightSpeed) {
}

// ==================== Arduino 入口 ====================
// 作用：初始化硬件资源。
// ==================================================
void setup() {
  Serial.begin(115200);
  pinMode(STATUS_LED_PIN, OUTPUT);
}

// ==================== 主循环 ====================
// 作用：持续处理通信、传感器和电机控制任务。
// ==================================================
void loop() {
  handleSerialCommand();
}
```

## Shell 脚本规范

Shell 脚本必须把环境检查、依赖安装、服务启动、日志输出、错误处理分开。危险操作前必须有清晰中文注释。

示例：

```bash
#!/usr/bin/env bash
# 作用：启动机器人上位机服务，包括前端、后端和 ROS 节点。

set -euo pipefail

# ==================== 路径配置 ====================
# 作用：定义脚本运行需要的项目目录和日志目录。
# ==================================================
PROJECT_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"
LOG_DIR="${PROJECT_DIR}/logs"

# ==================== 环境检查 ====================
# 作用：确认运行服务前必需的命令和目录存在。
# ==================================================
check_environment() {
  command -v pnpm >/dev/null
  command -v go >/dev/null
}

# ==================== 服务启动 ====================
# 作用：按顺序启动前端、后端和 ROS 相关进程。
# ==================================================
start_services() {
  mkdir -p "${LOG_DIR}"
}

# ==================== 脚本入口 ====================
# 作用：执行启动流程。
# ==================================================
main() {
  check_environment
  start_services
}

main "$@"
```

## API 放置规则

1. 通用 HTTP 客户端、鉴权、错误处理、重试逻辑放到公共 API 文件。
2. 模块专用接口放在模块目录内，例如机器人控制 API 放在 `robot` 或 `device` 模块下。
3. 页面组件不直接散写 `fetch`，必须通过对应 API 函数调用。
4. WebSocket、串口、ROS topic 这类连接型 API 必须单独封装连接、发送、接收、断线重连。
5. API 函数名必须体现业务动作，例如 `sendMoveCommand`、`fetchDeviceStatus`、`startObstacleDetection`。

## 目录命名规则

推荐按业务和职责分目录：

```text
frontend/
├── components/          # 通用组件
├── features/            # 业务模块
├── api/                 # 通用 API
├── hooks/               # 通用状态逻辑
├── styles/              # 全局样式
└── utils/               # 通用工具函数

backend/
├── modules/             # 业务模块
├── api/                 # 路由和接口
├── services/            # 业务服务
├── repositories/        # 数据访问
└── utils/               # 通用工具函数
```

## 提交前检查

提交代码前必须检查：

1. 新增文件开头是否有中文作用说明。
2. 功能代码是否按注释块分组。
3. 按钮、滑块、输入框等 UI 控件是否集中放置。
4. 事件处理和业务函数是否分开。
5. 通用 API 是否放到公共位置。
6. 模块专用 API 是否放在对应模块。
7. 是否存在命名模糊、重复代码、超长函数或一个文件塞太多功能的问题。
