# ORbot 项目代码风格规范

## 核心原则

本项目代码应该先像一份可读的系统结构说明，再像一份具体实现。

每个文件都应该让阅读者快速看明白：

- 这个文件主要负责什么。
- 当前模块有哪些按钮、面板、输入框、滑块、菜单、WebSocket、HTTP API、ROS topic、串口、定时任务或硬件控制入口。
- 每个入口分别负责什么交互或设备动作。
- 哪些 API 只属于某个具体模块。
- 哪些逻辑是通用能力，应该放到代码底部或公共文件中统一管理。

优先保证结构清晰，不追求把代码压得很短。避免把多个职责横向塞进一长行。

## 文件开头注释

每个源码文件开头都必须有中文注释框，说明这个文件的主要职责。

除非用户明确要求，或者文件职责发生真实变化，否则不要随意改动文件级注释。

Vue 文件使用这个格式：

```vue
<!--
|--------------------------------------------------------------------------
| 文件模块名称
|--------------------------------------------------------------------------
| 用中文说明这个 Vue 文件主要负责什么界面、哪些控件、服务哪个业务模块。
|--------------------------------------------------------------------------
-->
```

C、C++、TypeScript、JavaScript 文件使用这个格式：

```c
/*
|--------------------------------------------------------------------------
| 文件模块名称
|--------------------------------------------------------------------------
| 用中文说明这个代码文件主要负责什么、服务哪个机器人功能或系统模块。
|--------------------------------------------------------------------------
*/
```

Python、Shell 文件使用这个格式：

```py
#|--------------------------------------------------------------------------
# 文件模块名称
#|--------------------------------------------------------------------------
# 用中文说明这个脚本主要负责什么、启动哪个节点或处理哪个设备任务。
#|--------------------------------------------------------------------------
```

## 模块注释框

重要代码模块前必须使用中文注释框说明职责。

适合添加模块注释的地方：

- 一组相关状态。
- 某个按钮、面板、滑块、输入框、菜单或开关。
- 某个 HTTP API、WebSocket、串口、ROS topic 或硬件控制入口。
- 某个 C 后端路由处理器。
- 某个 ROS 节点的参数、发布订阅、模型推理、设备读写。
- 某个固件的引脚配置、协议解析、电机控制、传感器读取。
- 一组可复用工具函数。

统一使用这个注释框：

```c
/*
|--------------------------------------------------------------------------
| 模块名称
|--------------------------------------------------------------------------
| 用中文说明这个模块负责什么，不要逐行解释语法。
|--------------------------------------------------------------------------
*/
```

Vue template 使用这个注释框：

```vue
<!--
|--------------------------------------------------------------------------
| 可见界面模块名称
|--------------------------------------------------------------------------
| 用中文说明这里放置哪些按钮、面板、列表或状态展示。
|--------------------------------------------------------------------------
-->
```

Python 和 Shell 使用这个注释框：

```py
#|--------------------------------------------------------------------------
# 模块名称
#|--------------------------------------------------------------------------
# 用中文说明这个模块负责什么，不要逐行解释语法。
#|--------------------------------------------------------------------------
```

## 前端 Vue 代码顺序

Vue 文件建议按这个顺序组织：

1. 文件开头注释。
2. `import` 引入。
3. 交互控件清单。
4. 界面状态。
5. 控件专属样式。
6. 控件专属事件和 API。
7. 不属于单个控件的功能逻辑。
8. 共用工具函数放到底部。
9. `<template>` 按可见 UI 模块分组。

在 `<script setup>` 顶部列出主要控件：

```js
/*
|--------------------------------------------------------------------------
| 交互控件清单
|--------------------------------------------------------------------------
| 登录按钮：提交账号密码并建立 HostSession。
| 设置面板：修改摄像头地址、串口绑定和语言。
| 远程桌面窗口：通过 /ws/vnc 显示上位机桌面。
| 主机终端窗口：通过 /ws/shell 打开真实 shell。
|--------------------------------------------------------------------------
*/
```

控件专属状态、样式、事件、API 尽量放在同一个代码区域。

```js
/*
|--------------------------------------------------------------------------
| 登录按钮
|--------------------------------------------------------------------------
| 负责提交账号密码，成功后刷新当前用户会话并连接 WebSocket。
|--------------------------------------------------------------------------
*/
const loginButtonClass = [
  "rounded-md",
  "bg-cyan-500",
  "px-4",
  "py-2",
  "text-sm",
  "font-semibold"
];

async function handleLoginButtonClick() {
  loginError.value = "";
  await requestLoginSession(loginUser.value, loginPassword.value);
  await hydrateCurrentUser();
}

/*
|--------------------------------------------------------------------------
| 登录按钮 API
|--------------------------------------------------------------------------
| 仅服务登录按钮，用于向 C 后端提交账号密码。
|--------------------------------------------------------------------------
*/
async function requestLoginSession(username, password) {
  return apiFetch("/api/auth/login", {
    method : "POST",
    body   : JSON.stringify({ username, password })
  });
}
```

## 前端模板结构

在 `<template>` 中，按照可见 UI 模块分组，并在每组前面加中文注释框。

交互元素必须优先竖向排列属性和事件：

```vue
<!--
|--------------------------------------------------------------------------
| 登录操作模块
|--------------------------------------------------------------------------
| 放置账号输入、密码输入和登录按钮。
|--------------------------------------------------------------------------
-->
<button
  :class   = "loginButtonClass"
  type     = "button"
  :disabled = "loginBusy"
  @click   = "handleLoginButtonClick"
>
  登录
</button>
```

避免：

```vue
<button class="rounded-md bg-cyan-500 px-4 py-2 text-sm font-semibold" type="button" @click="handleLoginButtonClick">登录</button>
```

## C 后端代码顺序

C 后端文件建议按这个顺序组织：

1. 文件开头注释。
2. `#include` 引入。
3. 常量配置。
4. 结构体定义。
5. 全局运行状态。
6. 通用字符串、路径、JSON、HTTP 工具。
7. 认证和会话模块。
8. 具体 HTTP API 模块。
9. WebSocket 模块。
10. 静态资源模块。
11. 路由分发。
12. 服务启动和 `main`。

每个 HTTP API 必须有独立模块注释：

```c
/*
|--------------------------------------------------------------------------
| 设置 API
|--------------------------------------------------------------------------
| 读写 hostpc-settings.json，保存摄像头地址和串口绑定，供前端、ROS 和脚本共用。
|--------------------------------------------------------------------------
*/
static void handle_settings(const http_request_t *req) {
    ...
}
```

C 代码必须遵守：

- 路由处理函数命名为 `handle_xxx`。
- 通用工具函数放在文件前半部分或公共文件，不插在业务 API 中间。
- 一个函数只做一个清晰动作；过长时拆成解析、校验、执行、响应。
- 复杂条件不要横向写成长串，拆成命名清楚的局部变量。
- 用户可见错误返回 JSON，内部错误可以写日志。

## ROS / Python 代码顺序

ROS Python 节点建议按这个顺序组织：

1. 文件开头注释。
2. import 引入。
3. ROS 参数和 topic 名称。
4. 节点状态。
5. 发布器和订阅器。
6. 传感器读取或模型推理。
7. 控制指令生成。
8. 设备专属 API。
9. 通用工具函数。
10. `main()`。

示例：

```py
#|--------------------------------------------------------------------------
# YOLO 障碍物检测节点
#|--------------------------------------------------------------------------
# 读取摄像头图像，执行障碍物检测，并发布调试图像和避障结果。
#|--------------------------------------------------------------------------

#|--------------------------------------------------------------------------
# ROS 发布订阅
#|--------------------------------------------------------------------------
# 初始化图像订阅、检测结果发布和调试图像发布。
#|--------------------------------------------------------------------------
def setup_ros_io():
    ...

#|--------------------------------------------------------------------------
# 模型推理
#|--------------------------------------------------------------------------
# 处理摄像头图像并返回障碍物框、置信度和避障方向。
#|--------------------------------------------------------------------------
def detect_obstacles(frame):
    ...
```

## 固件 / C++ 代码顺序

固件代码必须把硬件资源和控制逻辑分开。

建议顺序：

1. 文件开头注释。
2. 库引入。
3. 引脚配置。
4. 通信协议常量。
5. 设备状态。
6. 串口 / I2C / CAN / PWM 初始化。
7. 指令解析。
8. 电机控制。
9. 传感器读取。
10. 安全保护。
11. `setup()`。
12. `loop()`。

示例：

```cpp
/*
|--------------------------------------------------------------------------
| 底盘电机控制固件
|--------------------------------------------------------------------------
| 解析上位机串口指令，通过驱动板控制全向轮底盘运动。
|--------------------------------------------------------------------------
*/

/*
|--------------------------------------------------------------------------
| 电机控制
|--------------------------------------------------------------------------
| 根据速度和方向计算电机输出，并写入驱动板。
|--------------------------------------------------------------------------
*/
void applyChassisVelocity(int vx, int vy, int wz) {
}
```

## Shell 脚本代码顺序

Shell 脚本建议按这个顺序组织：

1. shebang。
2. 文件开头注释。
3. `set -euo pipefail`。
4. 路径配置。
5. 环境检查。
6. 构建步骤。
7. 服务启动。
8. 日志和清理。
9. `main "$@"`。

危险操作前必须有中文注释说明影响范围。

```bash
#|--------------------------------------------------------------------------
# 服务启动
#|--------------------------------------------------------------------------
# 编译 C 后端，并以 8080 端口启动 HostPC 服务。
#|--------------------------------------------------------------------------
start_hostpc_backend() {
  ...
}
```

## API 归属规则

如果某个 API 只服务一个具体控件或模块，就把 API 包装函数放在该模块附近。

例如：

- 登录按钮 API 放在登录按钮模块附近。
- 设置面板 API 放在设置面板模块附近。
- C 后端 `/api/settings` 处理器放在设置 API 模块。
- ROS 串口读取函数放在对应串口桥模块。

如果某个 API 被多个控件或多个模块复用，就放到底部或公共文件：

- `apiFetch`
- JSON 转义
- Cookie 解析
- WebSocket 握手
- 路径清理
- ROS 消息格式转换

不要把通用工具函数插在某个按钮和该按钮事件之间，除非这个工具函数只属于这个按钮。

## 代码对齐和可读性

必须遵守：

- 很长的 Tailwind class 拆成数组。
- 交互元素的属性和事件一行写一个。
- 相邻的简单赋值可以适当对齐。
- C 结构体、常量和路由表保持竖向可读。
- Shell 命令不要写成难读的一长串，拆成函数。
- Python 节点不要把参数读取、模型推理、发布消息塞进同一个函数。

推荐：

```c
static const char *status_text(int status) {
    switch (status) {
        case 200: return "OK";
        case 401: return "Unauthorized";
        case 404: return "Not Found";
        default:  return "OK";
    }
}
```

避免：

```c
static const char *status_text(int status){if(status==200)return"OK";if(status==401)return"Unauthorized";return"OK";}
```

## 底部放置规则

这些内容应该放在文件底部或公共工具文件：

- 通用格式化函数。
- 纯工具函数。
- 共用校验函数。
- 共用 HTTP API 包装。
- 共用 WebSocket 工具。
- 不属于某个可见控件或具体设备的常量。

这些内容不应该插在某个控件、路由、ROS 节点或硬件控制模块中间。

## 验证要求

修改代码后必须按改动范围验证：

1. 改 Vue 前端：运行 `pnpm build`。
2. 改 C 后端：进入 `software/backend` 运行 `make`，并检查 `/api/health`。
3. 改 WebSocket：至少验证 `/ws`、`/ws/shell` 或 `/ws/vnc` 握手。
4. 改 ROS 节点：运行对应 Python 语法检查或 launch 文件检查。
5. 改 Shell：运行 `bash -n 脚本名`。
6. 当前目录是 git 仓库时，按项目要求处理 pull 和 push。
